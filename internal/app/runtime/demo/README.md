# demo runtime

Пакет `demo` содержит временную runtime-реализацию ботов для локальной разработки.

Он нужен не как “настоящий движок”, а как удобная заглушка, пока платформа
ещё не подключена к реальному Telegram runtime.

---

## Зачем нужен этот пакет

Проект развивается не как просто менеджер ботов, а как **e-commerce платформа**
с несколькими поверхностями управления и продаж:

- **Telegram** как клиентский и операторский канал
- **TUI** как терминальный интерфейс для контроля и управления
- **Web** как будущая админка магазина и ботов
- в будущем возможны и другие surface-layer интерфейсы

Чтобы развивать TUI, bootstrap и lifecycle-логику независимо от Telegram API,
нужен простой runtime, который можно быстро запускать локально.

Именно для этого существует пакет `demo`.

---

## Ответственность пакета

`demo` отвечает за одно:

- предоставить реализацию `manager.Runner` для локальной разработки

---

## Чего пакет не делает

Пакет `demo` не должен:

- содержать бизнес-логику магазина
- знать про товары, заказы, пользователей и продажи
- собирать приложение
- настраивать инфраструктуру
- запускать TUI
- подменять собой будущий Telegram runtime

Это именно **временная runtime-реализация**, а не постоянный центр системы.

---

## Поведение demo runner

Текущее поведение специально упрощено:

- `broken-bot` падает при старте
- `slow-bot` стартует с задержкой
- остальные боты считаются успешно запущенными сразу

Этого достаточно, чтобы разрабатывать и проверять:

- bootstrap enabled ботов
- статусы в manager
- отображение состояний в TUI
- обработку ошибок запуска
- сценарии локальной разработки без внешних интеграций

---

## Почему он вынесен в отдельный пакет

Раньше demo runner жил прямо в `cmd/tui/main.go`.

Для первого шага это было нормально, но по мере роста проекта entrypoint начал
брать на себя слишком много ответственности.

Вынос в отдельный пакет даёт несколько плюсов:

- `cmd/tui` остаётся entrypoint, а не местом для runtime-логики
- demo runtime можно заменить без переписывания `main.go`
- архитектурные роли становятся понятнее
- проект лучше готовится к реальному Telegram runtime

---

## Связь с manager

Пакет `demo` реализует контракт:

```go
type Runner interface {
    Run(ctx context.Context, spec BotSpec, ready func()) error
}
```

То есть demo runtime полностью подчиняется lifecycle-модели `manager`:

- manager передаёт cancelable context
- runtime сообщает о готовности через `ready`
- runtime завершается по отмене контекста или при ошибке

---

## Где используется

Сейчас demo runtime используется в TUI-срезе приложения:

- `cmd/tui`
- `internal/app/tuiapp`
- `internal/app/bootstrap`

Он помогает развивать терминальный интерфейс и orchestration-слой до того,
как будет подключён реальный Telegram runtime.

---

## Ограничения

Этот пакет не должен разрастаться.

Если появляется логика реальной интеграции:
- Telegram API
- обработка update
- polling/webhook
- авторизация
- реальные ошибки сети
- реальные команды и сценарии

то такая логика должна идти в отдельный пакет, например
`internal/app/runtime/telegram`.

---

## Будущее развитие

Следующий естественный шаг после `demo`:

- сделать реальный Telegram runtime
- сохранить совместимость с `manager.Runner`
- переключить `cmd/tui` и `tuiapp` на новый runtime без ломки остальных слоёв

В этом и была цель выноса `demo`:
подготовить чистую точку замены временного движка на настоящий.

---

## Итог

Пакет `demo` — это учебный и инфраструктурный runtime для локальной разработки.

Он помогает развивать:

- TUI
- bootstrap
- manager
- запуск enabled ботов

без привязки к внешним Telegram-зависимостям.

Когда реальный Telegram runtime будет готов, `demo` останется полезной
заглушкой для локальных сценариев, тестовых запусков и безопасной отладки.
